# Codex Development Rules

이 파일은 RuneGate Defense 저장소에서 Codex가 작업할 때 따라야 하는 개발 규칙이다. 저장소 루트 기준으로 전체 프로젝트에 적용한다.

## 기본 원칙

- 답변과 커밋 메시지는 기본적으로 한국어로 작성한다.
- 사용자가 별도 기준 문서를 지정하면, 이전 대화 맥락보다 해당 문서를 우선한다.
- 작업 시작 전 관련 문서를 먼저 읽는다. 특히 v0.x 작업은 루트의 작업 계획 문서와 `docs/` 안의 해당 태스크 문서를 source of truth로 삼는다.
- 구현은 기존 아키텍처와 네임스페이스 `RuneGate`를 유지한다.
- compile-safe를 최우선으로 한다. Unity Play Mode 진입을 막는 컴파일 오류를 남기지 않는다.
- 사용자가 명시하지 않은 최종 아트, 외부 서비스, 상용화 기능은 추가하지 않는다.

## 금지 사항

다음은 명시 요청이 없는 한 추가하지 않는다.

- 광고
- 인앱결제
- 로그인
- 서버 코드
- Firebase
- Addressables
- 외부 패키지
- 유료 에셋
- 가챠
- 멀티플레이
- 랭킹
- 클라우드 저장
- 분석/트래킹 SDK
- 최종 일러스트 대량 적용

## Unity 작업 규칙

- Unity 6 기준으로 작업한다.
- ScriptableObject 원본 값을 런타임 중 직접 변경하지 않는다.
- 데이터는 가능한 한 `ScriptableObject`, 저장은 로컬 JSON 방식을 유지한다.
- 새 폴더나 Unity 에셋을 만들면 관련 `.meta` 파일도 함께 관리한다.
- Audio, UI, Physics 같은 Unity 내장 모듈이 비활성일 수 있으므로 직접 타입 참조를 추가하기 전에 현재 프로젝트 컴파일 조건을 확인한다.
- 외부 패키지 없이 placeholder sprite, primitive object, IMGUI/기존 UI 기반으로 프로토타입을 유지한다.
- Bootstrap 메뉴는 기존 메뉴를 삭제하지 말고 새 버전 메뉴를 추가하거나 확장한다.
- Validator는 새로 추가된 필수 폴더, 스크립트, 데이터, 씬, 문서를 검사하도록 함께 갱신한다.

## 코드 스타일

- `private serialized field`를 선호하고, public mutable field는 피한다.
- MonoBehaviour는 작고 목적이 분명하게 유지한다.
- 전투 로직, 데이터, UI, 저장 로직의 책임을 섞지 않는다.
- null guard와 명확한 `Debug.LogWarning`을 추가한다.
- 복잡한 동작에만 짧은 주석을 남긴다.
- 기존 파일의 스타일과 패턴을 우선한다.

## Git 규칙

- 기본 브랜치 prefix는 `codex/`를 사용한다. 사용자가 브랜치명을 지정하면 그대로 따른다.
- 사용자가 "반영"을 요청하면 현재 작업 브랜치에 스테이징, 커밋, 푸시한다.
- 커밋 메시지는 한국어로 작성한다.
- 관련 없는 로컬 변경은 스테이징하지 않는다.
- 다음 파일/폴더는 사용자가 명시하지 않는 한 커밋하지 않는다.
  - `My project/`
  - 임시 PDF
  - 프롬프트/클립보드 임시 파일
  - 사용자 또는 Unity Editor가 별도로 수정한 씬 파일
  - 목적이 불명확한 `ProjectSettings` 변경
- 이미 더러운 작업 트리가 있을 수 있으므로, 내가 만든 변경과 기존 변경을 반드시 분리해서 다룬다.
- `git reset --hard`, `git checkout -- <file>` 같은 되돌리기 명령은 사용자가 명확히 요청한 경우에만 사용한다.

## 검증 규칙

가능한 경우 다음을 확인한다.

- `git diff --check`
- C# 컴파일
- Unity 메뉴 기반 Validator
- Bootstrap 메뉴 실행 후 샘플 씬 수동 테스트

Unity Editor가 열려 있어 batchmode가 막히면 그 사실을 보고하고, 최소한 로컬 C# 컴파일과 로그 확인을 수행한다.

## 보고 형식

구현 후에는 사용자가 요청한 항목을 우선 보고한다. 별도 지정이 없으면 다음을 짧게 정리한다.

- 생성/수정 파일
- 실행 방법
- 수동 테스트 방법
- 남은 한계점
- 실제 검증 여부

컴파일 오류를 고친 경우에는 원인, 수정 내용, 확인 결과를 먼저 말한다.
