You are working on a Unity 6 mobile 2D defense game project.

Project name:
RuneGate Defense

Game concept:
A vertical 2D roguelite fantasy defense game. Monsters invade through a Rune Gate, and the player defends a Kingdom Crystal by placing heroes in 3 lanes. Heroes auto-attack, the player can manually trigger skills, and after each wave the player chooses 1 of 3 rune cards. The MVP must run locally without server, login, ads, IAP, gacha, or external paid APIs.

Primary goals:
1. Set up a maintainable Unity project structure.
2. Implement the initial battle prototype architecture.
3. Prepare the project for future expansion with heroes, monsters, skills, runes, stages, and waves.
4. Use clean C# code suitable for long-term maintenance.
5. Use ScriptableObject-based data design where appropriate.
6. Avoid hardcoding game balance values directly in MonoBehaviour logic.
7. Do not add ads, billing, networking, login, analytics, Firebase, or Addressables yet.
8. Do not import any paid assets or external packages.

Technology assumptions:
- Unity 6
- C#
- 2D mobile portrait game
- Local save only
- No server
- No gacha
- No paid API
- No OCR
- No AI API inside the game

Target MVP prototype:
- 3-lane battle field
- Kingdom Crystal HP
- Monster spawning by waves
- Hero auto attack
- Manual hero skill trigger hooks
- Rune card selection after each wave
- Victory/defeat handling
- Local architecture ready for later content expansion

3. Runtime battle components

Create:
- BattleManager.cs
- LaneManager.cs
- CrystalController.cs
- HeroController.cs
- MonsterController.cs
- ProjectileController.cs
- WaveManager.cs
- RuneManager.cs

- BattleResult.cs

BattleManager responsibilities:
- Own battle state
- Initialize stage
- Start wave
- Detect wave clear
- Trigger rune selection
- Handle victory and defeat
- Communicate with UI through C# events
- Avoid direct UI logic inside core battle logic

LaneManager responsibilities:
- Provide lane spawn positions and crystal target positions
- Validate lane index
- Support 3 lanes by default

CrystalController responsibilities:
- Store current HP and max HP
- Take damage when monsters reach the crystal
- Raise event when HP changes
- Raise event when destroyed

HeroController responsibilities:
- Initialize from HeroData
- Store runtime HP and stats
- Find target within range
- Auto attack based on attackSpeed
- Trigger SkillController when manual skill is requested
- Keep logic generic so new HeroData can be added without new scripts

MonsterController responsibilities:
- Initialize from MonsterData
- Move along assigned lane toward crystal
- Take damage
- Die and notify WaveManager/BattleManager
- Damage crystal when reaching destination
- Support simple special type hooks but do not fully implement every monster ability yet

ProjectileController responsibilities:
- Move toward target
- Apply damage on hit
- Handle null target safely

WaveManager responsibilities:
- Spawn monsters according to StageData waves
- Track alive monsters and remaining spawns
- Notify BattleManager when wave is complete

RuneManager responsibilities:
- Select 3 rune cards from available RuneData list
- Prevent duplicate choices in the same selection
- Notify BattleManager when a rune is selected

RuneEffectApplier responsibilities:
- Apply rune effects to runtime battle context
- Support at least these effect keys:
  - hero_attack_percent
  - hero_attack_speed_percent
  - crystal_heal_flat
  - crystal_shield_flat placeholder
  - monster_slow_percent placeholder
  - skill_cooldown_percent
  - boss_damage_percent placeholder

SkillController responsibilities:
- Hold SkillData
- Track cooldown
- Expose CanUseSkill
- Provide UseSkill method
- Implement simple placeholder behavior:
  - direct damage skill
  - area damage placeholder
  - heal placeholder

7. Coding standards

Follow these rules:
- Use namespaces: RuneGate
- Keep MonoBehaviours small and focused.
- Use events where UI needs to observe battle state.
- Use serialized private fields instead of public mutable fields where appropriate.
- Add null guards and clear Debug.LogWarning messages.
- Do not use singletons unless strongly justified.
- Avoid deeply coupled references.
- Avoid magic numbers in gameplay logic.
- Use ScriptableObjects for balance data.
- Keep prototype code simple but not messy.
- Add comments only where they clarify non-obvious logic.
- Make the project compile.
